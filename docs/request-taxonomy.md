# Request Taxonomy

## Purpose

A routing system is only as good as the categories it can recognize.

This taxonomy is intended to define the major classes of work `ai-it-team` should be able to receive.

## Core request classes

### 1. Support issue
User-facing problem or request for assistance.

Examples:
- cannot connect to VPN
- email not syncing
- workstation issue

### 2. Access request
A request to add, remove, change, or validate access.

Examples:
- permission request
- role change
- account access issue

### 3. Incident / outage
Urgent service degradation or broad operational failure.

Examples:
- internal app unavailable
- multiple users affected
- alert indicates major failure

### 4. Infrastructure change
A proposed change to systems, cloud, network, or operational platform.

Examples:
- server config change
- firewall rule change
- patching plan

### 5. Security review
A question or task requiring explicit risk or security posture review.

Examples:
- suspicious login assessment
- risky exception request
- privileged access concern

### 6. Documentation task
A request to create, refine, or update knowledge artifacts.

Examples:
- runbook update
- procedure creation
- post-incident documentation

### 7. Monitoring / alert response
A machine-generated signal requiring interpretation or triage.

Examples:
- host alert
- service degradation alert
- anomaly notification

## Principle

The taxonomy should support better routing, not taxonomy theater.
Categories should stay broad enough to remain usable.
