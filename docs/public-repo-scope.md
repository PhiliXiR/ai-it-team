# Public Repo Scope

## What belongs here

- team structure
- role definitions
- workflow design
- examples using fake or generic infrastructure
- public-safe implementation ideas
- documentation templates
- architecture notes

## What does not belong here

- real infrastructure names
- secrets or credentials
- private diagrams
- internal runbooks copied from real environments
- real incident records from private systems

## Why this matters

A public repo should be useful, shareable, and safe.
It should demonstrate the design of the system without leaking the environment of whoever builds it.
